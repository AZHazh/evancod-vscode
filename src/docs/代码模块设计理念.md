## 1. 整体架构设计

### 1.1 分层架构理念

```
┌─────────────────────────────────────┐
│         VSCode Extension Host        │  ← Node.js 环境
│  ┌───────────────────────────────┐  │
│  │  Services（业务服务层）        │  │
│  │  - ChatService                │  │
│  │  - ProviderService            │  │
│  │  - WebviewManager             │  │
│  └───────────────────────────────┘  │
│  ┌───────────────────────────────┐  │
│  │  Adapters（适配器层）          │  │
│  │  - FileSystemAdapter          │  │
│  │  - ConfigAdapter              │  │
│  │  - StorageAdapter             │  │
│  └───────────────────────────────┘  │
└─────────────────────────────────────┘
                  ↕ postMessage
┌─────────────────────────────────────┐
│         Webview (Vue 3 应用)         │  ← 浏览器环境
│  ┌───────────────────────────────┐  │
│  │  Views（视图层）               │  │
│  │  - ChatView                   │  │
│  └───────────────────────────────┘  │
│  ┌───────────────────────────────┐  │
│  │  Components（组件层）          │  │
│  │  - TopBar, MessageList, etc.  │  │
│  └───────────────────────────────┘  │
│  ┌───────────────────────────────┐  │
│  │  Stores（状态层 - Pinia）     │  │
│  │  - chatStore                  │  │
│  │  - providerStore              │  │
│  └───────────────────────────────┘  │
└─────────────────────────────────────┘
```

**设计原则**:
1. **关注点分离** - Extension 处理业务逻辑，Webview 只负责 UI
2. **单向数据流** - Extension 是数据源，Webview 通过消息请求数据
3. **类型安全** - TypeScript 贯穿整个项目
4. **可测试性** - 每个模块都可以独立测试

## 2. Extension Host 层设计

### 2.1 extension.ts - 入口文件

**职责**:
- 插件生命周期管理（activate/deactivate）
- 服务初始化和依赖注入
- 命令注册
- 资源清理

**设计模式**: **依赖注入 + 服务定位器**

```typescript
// 为什么这样设计？
// 1. 集中管理所有服务的创建和依赖关系
// 2. 便于测试（可以注入 mock 服务）
// 3. 清晰的初始化顺序

export async function activate(context: vscode.ExtensionContext) {
  // 1. 先初始化基础服务（无依赖）
  providerService = new ProviderService(context)
  await providerService.initialize()

  // 2. 再初始化依赖基础服务的服务
  chatService = new ChatService(context, providerService)
  webviewManager = new WebviewManager(context, chatService)
  
  // 3. 最后初始化 UI 服务
  statusBarService = new StatusBarService(context, providerService)
}
```

**为什么不用全局变量？**
- 使用模块级变量（let）而不是全局变量，保证作用域隔离
- 通过 ExtensionContext.subscriptions 管理资源生命周期
- deactivate 时自动清理

---

### 2.2 ChatService - 会话管理

**职责**:
- 管理会话列表和当前会话
- 处理用户消息
- 调用 QueryEngine（Phase 2）
- 持久化会话数据

**设计模式**: **仓储模式 (Repository Pattern)**

```typescript
// 为什么使用仓储模式？
// 1. 封装数据访问逻辑
// 2. 隔离业务逻辑和存储实现
// 3. 便于切换存储方式（内存 → 文件 → 数据库）

class ChatService {
  private sessions: Session[] = []        // 内存中的数据
  private currentSessionId: string | null // 当前活动会话
  
  // 查询方法
  getCurrentSession(): Session | null
  getSessions(): Session[]
  
  // 命令方法
  createNewSession(): Session
  async sendMessage(content: string): Promise<void>
}
```

**为什么不直接操作文件？**
- 先在内存中操作，响应快
- 批量写入文件，减少 I/O
- 便于实现撤销/重做功能

---

### 2.3 ProviderService - Provider 管理

**职责**:
- 管理 API Provider 配置
- 读写 `~/.claude/cc-evancod/providers.json`
- 激活/切换 Provider
- 与 Desktop 版本配置兼容

**设计模式**: **单例模式 + 配置管理器**

```typescript
// 为什么使用单例？
// 1. Provider 配置是全局唯一的
// 2. 避免多次读取同一个文件
// 3. 保证配置一致性

class ProviderService {
  private providers: Provider[] = []
  private activeProviderId: string | null
  private configPath: string
  
  // 配置文件路径固定，与 Desktop 兼容
  constructor() {
    this.configPath = path.join(
      os.homedir(), 
      '.claude', 
      'cc-evancod', 
      'providers.json'
    )
  }
}
```

**为什么配置文件在 ~/.claude？**
- 与 Desktop 版本共享配置
- 用户数据在用户目录，不受插件更新影响
- 符合 Unix/Linux 惯例

### 2.4 WebviewManager - Webview 生命周期

**职责**:
- 创建和销毁 Webview
- 加载 HTML 内容
- 处理消息通信
- 区分开发/生产模式

**设计模式**: **工厂模式 + 单例模式**

```typescript
// 为什么只创建一个 Webview？
// 1. VSCode Webview 比较重，不适合创建多个
// 2. 用户一次只能看一个聊天界面
// 3. 保持状态（retainContextWhenHidden: true）

class WebviewManager {
  private panel: vscode.WebviewPanel | undefined
  
  show() {
    if (this.panel) {
      this.panel.reveal()  // 复用已有的
      return
    }
    this.panel = vscode.window.createWebviewPanel(...)
  }
}
```

**开发模式 vs 生产模式**:
```typescript
// 开发模式：连接到 Vite dev server（热重载）
if (isDevelopment) {
  return `<script src="http://localhost:5173/src/main.ts"></script>`
}

// 生产模式：使用打包后的文件
const scriptUri = webview.asWebviewUri(...)
```

## 3. Webview 层设计

### 3.1 Vue 3 架构选择

**为什么选择 Vue 3？**
1. **Composition API** - 更好的逻辑复用
2. **响应式系统** - 自动追踪依赖，减少手动更新
3. **单文件组件** - 结构清晰，便于维护
4. **TypeScript 支持** - 类型安全

**为什么用 Pinia 而不是 Vuex？**
1. **更简单的 API** - 不需要 mutations
2. **更好的 TypeScript 支持** - 自动类型推导
3. **Vue 3 官方推荐**

---

### 3.2 组件设计原则

#### 3.2.1 组件拆分粒度

```
ChatView (页面级)
├── TopBar (布局级)
│   ├── NewSessionButton (功能级)
│   └── SyncButton (功能级)
├── MessageList (布局级)
│   └── MessageItem (功能级)
└── ChatInput (布局级)
    ├── InputTextarea (功能级)
    └── InputControls (功能级)
```

**拆分原则**:
1. **单一职责** - 每个组件只做一件事
2. **可复用性** - 独立的功能组件可以复用
3. **可测试性** - 小组件更容易测试

#### 3.2.2 Props vs Store

**何时使用 Props？**
- 父子组件直接通信
- 数据简单且不需要跨层级传递
- 组件需要复用（不依赖全局状态）

**何时使用 Store？**
- 多个组件需要共享数据
- 数据需要持久化或来自 Extension
- 复杂的状态管理逻辑

```vue
<!-- ❌ 不好的设计：所有数据都通过 Props -->
<MessageItem :message="msg" :user="user" :config="config" />

<!-- ✅ 好的设计：组件数据来自 Props，全局数据来自 Store -->
<MessageItem :message="msg" />
<!-- 组件内部使用 chatStore.currentUser, chatStore.config -->
```

---

### 3.3 状态管理设计（Pinia）

#### 3.3.1 chatStore 设计

```typescript
// 为什么这样设计状态？
export const useChatStore = defineStore('chat', () => {
  // State - 响应式数据
  const currentSession = ref<Session | null>(null)
  const sessions = ref<Session[]>([])
  const isStreaming = ref(false)
  
  // Getters - 计算属性
  const messages = computed(() => 
    currentSession.value?.messages || []
  )
  
  // Actions - 方法
  function sendMessage(content: string) {
    // 发送到 Extension
    vscode.postMessage({ type: 'chat.send', data: { content } })
  }
  
  return { currentSession, messages, sendMessage }
})
```

**设计要点**:
1. **ref vs reactive** - 使用 ref 便于类型推导
2. **computed** - 缓存计算结果，避免重复计算
3. **方法封装** - 隐藏 postMessage 细节

#### 3.3.2 状态持久化

```typescript
// Webview 的状态不需要持久化到文件
// 因为 Extension 是数据源，Webview 只是展示
// 每次打开 Webview，从 Extension 恢复状态

function initialize() {
  window.addEventListener('message', handleMessage)
  
  // 通知 Extension 已准备好
  vscode.postMessage({ type: 'ready' })
  
  // Extension 会回复初始数据
}
```

## 4. 通信机制设计

### 4.1 消息协议设计

**为什么需要类型化的消息？**
```typescript
// ❌ 不好的设计：任意对象
vscode.postMessage({ cmd: 'send', text: 'hello' })

// ✅ 好的设计：类型化消息
interface ChatSendMessage {
  type: 'chat.send'
  data: { content: string; images?: File[] }
}

vscode.postMessage({
  type: 'chat.send',
  data: { content: 'hello' }
})
```

**好处**:
1. **类型安全** - TypeScript 编译时检查
2. **易于维护** - 清楚知道有哪些消息类型
3. **便于调试** - 根据 type 字段快速定位

---

### 4.2 消息流向设计

#### Extension → Webview (推送模式)
```typescript
// Extension 主动推送数据
webview.postMessage({
  type: 'chat.message.stream',
  data: { content: 'Hello', delta: true }
})
```

**适用场景**:
- 流式响应（AI 回复）
- 状态变化通知
- 错误提示

#### Webview → Extension (请求-响应模式)
```typescript
// Webview 请求数据
vscode.postMessage({ type: 'provider.list' })

// Extension 响应
webview.postMessage({
  type: 'provider.list.response',
  data: { providers: [...] }
})
```

**适用场景**:
- 用户操作（发送消息、创建会话）
- 查询数据（获取 Provider 列表）

---

### 4.3 错误处理设计

```typescript
// 统一的错误消息格式
interface ErrorMessage {
  type: 'error'
  data: {
    message: string
    code?: string
    context?: any
  }
}

// Extension 捕获错误后发送
try {
  await chatService.sendMessage(content)
} catch (error) {
  webview.postMessage({
    type: 'error',
    data: {
      message: error.message,
      code: 'SEND_MESSAGE_FAILED'
    }
  })
}
```

## 5. 状态管理设计

### 5.1 状态所有权

**规则**: Extension 拥有所有业务状态，Webview 只有 UI 状态

```
Extension 状态:                Webview 状态:
- Session 列表                 - 滚动位置
- Provider 配置                - 输入框内容（未发送）
- 消息历史                     - 模态框打开/关闭
- 当前模型                     - 下拉菜单展开状态
```

**为什么这样设计？**
1. **单一数据源** - 避免数据不一致
2. **Webview 可重建** - 关闭后状态丢失不影响
3. **便于调试** - 业务状态都在 Extension

---

### 5.2 状态同步策略

```typescript
// 1. Webview 启动时，Extension 推送初始状态
case 'ready':
  webview.postMessage({
    type: 'session.restored',
    data: {
      session: chatService.getCurrentSession(),
      sessions: chatService.getSessions()
    }
  })

// 2. 状态变化时，Extension 推送更新
chatService.on('sessionUpdated', (session) => {
  webview.postMessage({
    type: 'session.updated',
    data: { session }
  })
})

// 3. Webview 只需要监听消息，更新本地状态
function handleMessage(event: MessageEvent) {
  const message = event.data
  if (message.type === 'session.updated') {
    currentSession.value = message.data.session
  }
}
```

## 6. 服务层设计

### 6.1 服务职责划分

| 服务 | 职责 | 依赖 |
|------|------|------|
| ChatService | 会话管理、消息处理 | ProviderService |
| ProviderService | Provider 配置 | 无 |
| WebviewManager | Webview 生命周期 | ChatService |
| StatusBarService | 状态栏 UI | ProviderService |

**为什么要分这么多服务？**
1. **单一职责** - 每个服务只做一件事
2. **可测试性** - 独立测试每个服务
3. **可扩展性** - 新增功能只需新增服务

---

### 6.2 依赖注入

```typescript
// ❌ 不好的设计：服务内部创建依赖
class ChatService {
  constructor() {
    this.providerService = new ProviderService()  // 耦合
  }
}

// ✅ 好的设计：依赖注入
class ChatService {
  constructor(
    private context: vscode.ExtensionContext,
    private providerService: ProviderService  // 注入
  ) {}
}

// 在 extension.ts 中组装
providerService = new ProviderService(context)
chatService = new ChatService(context, providerService)
```

**好处**:
- 测试时可以注入 Mock 对象
- 服务之间低耦合
- 便于修改实现

## 7. 适配器模式

### 7.1 为什么需要适配器？

VSCode API 和标准 Node.js API 不同：

```typescript
// Node.js 标准 API
import * as fs from 'fs'
const content = fs.readFileSync('/path/to/file', 'utf-8')

// VSCode API
const uri = vscode.Uri.file('/path/to/file')
const content = await vscode.workspace.fs.readFile(uri)
const text = Buffer.from(content).toString('utf-8')
```

**适配器的作用**:
- 统一接口
- 隔离平台差异
- 便于测试（可以 Mock 适配器）

---

### 7.2 FileSystemAdapter 设计

```typescript
// 定义统一的接口
interface IFileSystemAdapter {
  readFile(path: string): Promise<string>
  writeFile(path: string, content: string): Promise<void>
  exists(path: string): Promise<boolean>
}

// VSCode 实现
class VSCodeFileSystemAdapter implements IFileSystemAdapter {
  async readFile(path: string): Promise<string> {
    const uri = vscode.Uri.file(path)
    const content = await vscode.workspace.fs.readFile(uri)
    return Buffer.from(content).toString('utf-8')
  }
}

// 测试时使用 Mock 实现
class MockFileSystemAdapter implements IFileSystemAdapter {
  private files = new Map<string, string>()
  
  async readFile(path: string): Promise<string> {
    return this.files.get(path) || ''
  }
}
```

## 8. 类型系统设计

### 8.1 共享类型

Extension 和 Webview 共享相同的类型定义：

```typescript
// src/types/index.ts (Extension)
export interface Message {
  id: string
  role: 'user' | 'assistant' | 'system'
  content: string
  timestamp: number
}

// webview/src/types/index.ts (Webview)
export interface Message {
  id: string
  role: 'user' | 'assistant' | 'system'
  content: string
  timestamp: number
}
```

**为什么重复定义？**
- Extension 和 Webview 是独立编译的
- 不能直接 import 对方的类型
- 通过文档保持一致性

**未来优化**: 使用 npm 包共享类型定义

---

### 8.2 类型安全的消息

```typescript
// 定义所有消息类型
type ExtensionMessage =
  | { type: 'chat.message.stream'; data: { content: string } }
  | { type: 'session.restored'; data: { session: Session } }
  | { type: 'error'; data: { message: string } }

// 类型守卫
function isStreamMessage(msg: ExtensionMessage): 
  msg is { type: 'chat.message.stream'; data: { content: string } } {
  return msg.type === 'chat.message.stream'
}

// 使用
if (isStreamMessage(message)) {
  // TypeScript 知道这里 message.data.content 存在
  console.log(message.data.content)
}
```

## 9. 设计模式总结

| 模式 | 使用场景 | 好处 |
|------|----------|------|
| 依赖注入 | 服务组装 | 低耦合、可测试 |

| 观察者模式 | 事件通信 | 解耦发送和接收 |
