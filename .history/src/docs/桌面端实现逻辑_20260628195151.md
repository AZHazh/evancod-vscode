# 服务商与多模型协议会话实现

本文梳理项目内「服务商」功能的实现方式，以及桌面会话如何兼容不同模型协议、不同 provider/model 的运行逻辑。

## 总览

项目把所有会话入口统一成 Anthropic Messages API 形态：

```text
桌面 UI 选择 provider/model
  ↓ WebSocket set_runtime_config
本地 server 记录会话级 runtime override
  ↓ 启动或重启 CLI 子进程
CLI 读取 ANTHROPIC_* / OpenAI OAuth 等运行时 env
  ↓
Anthropic 兼容服务：直连 /v1/messages
OpenAI Chat/Responses：先打本地 proxy，再转换到上游协议
```

核心思想是：CLI/agent-loop 仍以 Anthropic 协议工作；非 Anthropic 协议通过本地 server 的 proxy 做请求/响应转换。

## 数据模型

服务商类型定义在 `src/server/types/provider.ts`，桌面端镜像类型在 `desktop/src/types/provider.ts`。

### Provider 核心字段

`SavedProvider` 保存以下关键信息：

- `id`：用户保存的 provider 实例 ID。
- `presetId`：来源预设，例如 custom、内置服务商预设等。
- `name`：桌面端显示名称。
- `apiKey`：上游密钥。
- `authStrategy`：Anthropic 直连时使用哪种鉴权头。
- `baseUrl`：上游 API 地址。
- `apiFormat`：协议格式，支持 `anthropic`、`openai_chat`、`openai_responses`。
- `runtimeKind`：运行时类型，支持普通 Anthropic 兼容 provider 与 OpenAI OAuth provider。
- `models`：`main`、`haiku`、`sonnet`、`opus` 四类模型映射。
- `autoCompactWindow` / `modelContextWindows`：上下文窗口和自动压缩相关运行参数。

### 内置 provider

`src/server/types/provider.ts` 定义了两个内置 provider ID：

- `claude-official`：Claude 官方模式。
- `openai-official`：ChatGPT/OpenAI 官方 OAuth 模式。

它们会和用户保存的 provider 一起参与排序与选择，但不完全走普通自定义 provider 的保存结构。

## 持久化与迁移

`ProviderService` 位于 `src/server/services/providerService.ts`，负责服务商的读取、写入、激活、测试和运行时环境构造。

### 存储位置

- provider 索引：`~/.claude/cc-evancod/providers.json`
- 桌面管理的 settings：`~/.claude/cc-evancod/settings.json`

这两个文件与原 Claude Code 的 `~/.claude/settings.json` 隔离，避免桌面端 provider 管理污染用户已有配置。

### 读取与规范化

`ProviderService.readIndex()` 会先调用持久化升级逻辑，再通过 `readRecoverableJsonFile()` 读取 provider 索引，并用 `normalizeProvidersIndex()` 做兼容处理。

`normalizeProvidersIndex()` 在 `src/server/services/providerRuntimeEnv.ts` 中实现，兼容点包括：

- 旧字段 `activeProviderId` 迁移到 `activeId`。
- 缺失 `apiFormat` 时默认 `anthropic`。
- 缺失 `runtimeKind` 时默认 `anthropic_compatible`。
- 过滤无效 provider。
- 补齐 `providerOrder`，并保留内置 provider 的显示顺序。

## 服务商 API

REST API 在 `src/server/api/providers.ts`：

| 接口                               | 作用                               |
| ---------------------------------- | ---------------------------------- |
| `GET /api/providers`               | 获取保存的 provider、当前 activeId |
| `GET /api/providers/presets`       | 获取可选预设                       |
| `GET /api/providers/auth-status`   | 检查当前是否有可用鉴权             |
| `GET /api/providers/settings`      | 读取桌面管理的 settings            |
| `PUT /api/providers/settings`      | 更新桌面管理的 settings            |
| `POST /api/providers`              | 新增 provider                      |
| `PUT /api/providers/:id`           | 更新 provider                      |
| `DELETE /api/providers/:id`        | 删除 provider，禁止删除当前激活项  |
| `POST /api/providers/:id/activate` | 激活指定 provider                  |
| `POST /api/providers/official`     | 切回 Claude 官方模式               |
| `PUT /api/providers/reorder`       | 保存 provider 显示顺序             |
| `POST /api/providers/:id/test`     | 测试已保存 provider                |
| `POST /api/providers/test`         | 测试未保存配置                     |

桌面端 API 封装在 `desktop/src/api/providers.ts`，状态管理在 `desktop/src/stores/providerStore.ts`。

## 桌面端服务商功能

设置页的 Provider UI 在 `desktop/src/pages/Settings.tsx` 的 `ProviderSettings` 和 `ProviderModal`。

### 主要能力

- 加载 provider 列表和预设。
- 新增、编辑、删除 provider。
- 拖拽排序 provider，包括内置 provider。
- 设置默认 provider。
- 测试 provider 连接。
- 配置协议格式 `apiFormat`。
- 配置 Anthropic 鉴权策略 `authStrategy`。
- 配置四类模型映射。
- 配置上下文窗口与自动压缩窗口。
- 同步 new-api 令牌到 provider（入口在设置页，具体服务在 `src/server/services/newApiSyncService.ts`）。

`providerStore.activateProvider()` 激活 provider 后，会同步刷新默认模型：

- `openai-official` 使用 OpenAI 官方默认模型。
- 自定义 provider 使用该 provider 的 `models.main`。
- 切回 Claude 官方模式使用 Claude 官方默认模型。

这样避免 settings 中残留旧 provider 的 model id，导致默认模型指向不存在的上游模型。

## Provider 激活与运行时环境

激活 provider 的核心在 `ProviderService.activateProvider()`：

1. 读取 provider 索引。
2. 校验 provider 是否存在，内置 OpenAI 官方 provider 走特殊分支。
3. 写入 `activeId`。
4. 根据 provider 类型同步运行时 env 到 `~/.claude/cc-evancod/settings.json`。

### Anthropic 兼容 provider

当 `apiFormat === 'anthropic'` 时，`buildProviderManagedEnv()` 会生成：

- `ANTHROPIC_BASE_URL = provider.baseUrl`
- `ANTHROPIC_MODEL = models.main`
- `ANTHROPIC_DEFAULT_HAIKU_MODEL = models.haiku`
- `ANTHROPIC_DEFAULT_SONNET_MODEL = models.sonnet`
- `ANTHROPIC_DEFAULT_OPUS_MODEL = models.opus`
- `ANTHROPIC_API_KEY` 或 `ANTHROPIC_AUTH_TOKEN`，取决于 `authStrategy`
- 可选 `CLAUDE_CODE_AUTO_COMPACT_WINDOW`
- 可选 `CLAUDE_CODE_MODEL_CONTEXT_WINDOWS`
- 可选 attribution header env

CLI 子进程随后直接向上游 `baseUrl/v1/messages` 发 Anthropic Messages 请求。

### OpenAI Chat / Responses provider

当 `apiFormat !== 'anthropic'` 时，provider 需要本地代理：

- `ANTHROPIC_BASE_URL` 被改成本地 server 地址，例如 `http://127.0.0.1:3456/proxy`。
- `ANTHROPIC_API_KEY` 写成 `proxy-managed`，避免 CLI 直接持有真实上游 key。
- 真实上游 `apiKey` 保留在 provider 存储，由 server proxy 使用。
- 模型映射仍通过 `ANTHROPIC_MODEL` 等 env 传给 CLI。

对于会话级显式 provider，`getProviderRuntimeEnv()` 会生成 `http://127.0.0.1:<port>/proxy/providers/<providerId>`，让每个会话可以绑定自己的 provider，而不是依赖全局 active provider。

### OpenAI 官方 OAuth provider

`runtimeKind === 'openai_oauth'` 时，`buildProviderManagedEnv()` 直接走 `buildOpenAIOfficialRuntimeEnv()`，通过 OpenAI OAuth 相关 env 告诉 CLI 使用官方 OpenAI 登录态。

ConversationService 还会在官方模式下通过 `buildOfficialOAuthEnv()` 注入：

- `CLAUDE_CODE_ENTRYPOINT=claude-desktop`
- 可用时注入 `CLAUDE_CODE_OAUTH_TOKEN`

这样可以让桌面端避开外部残留 `ANTHROPIC_*` env，并使用 Evancod 自管 OAuth token。

## 协议转换代理

代理入口在 `src/server/proxy/handler.ts`。

### 支持的代理路径

- `POST /proxy/v1/messages`：使用当前 active provider。
- `POST /proxy/providers/:providerId/v1/messages`：使用显式 provider，支持会话级 provider 隔离。

代理只处理 Anthropic Messages 请求入口，因为 CLI 始终按 Anthropic 协议调用。

### openai_chat 转换链路

`apiFormat === 'openai_chat'`：

1. 接收 Anthropic request。
2. `anthropicToOpenaiChat()` 转成 OpenAI Chat Completions 请求。
3. 请求上游 `baseUrl/v1/chat/completions`。
4. 非流式响应用 `openaiChatToAnthropic()` 转回 Anthropic message。
5. 流式响应用 `openaiChatStreamToAnthropic()` 转成 Anthropic SSE。

代理中还包含 DeepSeek/opencode 兼容逻辑：针对 reasoning 内容、thinking toggle、图片内容降级为 text-only 等做适配。

### openai_responses 转换链路

`apiFormat === 'openai_responses'`：

1. 接收 Anthropic request。
2. `anthropicToOpenaiResponses()` 转成 OpenAI Responses 请求。
3. 请求上游 `baseUrl/v1/responses`。
4. 非流式响应用 `openaiResponsesToAnthropic()` 转回 Anthropic message。
5. 流式响应用 `openaiResponsesStreamToAnthropic()` 转成 Anthropic SSE。

Responses 代理会结合 `resolvePromptCacheKey()` 生成 cache key，用于上游 prompt cache 相关参数。

### 代理的辅助能力

代理还负责：

- 标准化模型名：`normalizeModelStringForAPI()`。
- 应用网络代理配置：`loadNetworkSettings()` + `getProxyFetchOptions()`。
- 非流式请求超时控制。
- 流式请求 header 超时与 idle timeout。
- API trace 采集：记录 Anthropic 请求、上游请求、上游响应、转换后的 Anthropic 响应。

## Provider 测试逻辑

`ProviderService.testProviderConfig()` 分两步测试：

1. `testConnectivity()`：直接打上游接口，验证 URL、key、model、响应结构。
2. `testProxyPipeline()`：仅 OpenAI 格式需要，模拟 CLI 的 Anthropic 请求，经转换后打上游，再转回 Anthropic 响应并校验。

不同 `apiFormat` 的直连测试地址不同：

- `anthropic` → `baseUrl/v1/messages`
- `openai_chat` → `baseUrl/v1/chat/completions`
- `openai_responses` → `baseUrl/v1/responses`

测试失败会记录 diagnostics 事件，便于设置页和诊断页排查。

## 模型列表与默认模型

模型 API 在 `src/server/api/models.ts`。

### 获取模型列表

`GET /api/models` 的优先级：

1. active provider 是 OpenAI 官方：返回 OpenAI Codex 模型目录。
2. active provider 是自定义 provider：返回该 provider 的四类模型映射去重后的列表。
3. 无 active provider：从环境变量读取 Anthropic 模型；若为空则返回 Claude 官方默认模型；同时补充 OpenAI OAuth 登录模型。

### 当前模型

`GET /api/models/current` 会根据 active provider 决定读取来源：

- active provider 或 OpenAI 官方：读取 `ProviderService.getManagedSettings()`。
- 无 active provider：读取普通用户 settings。

`PUT /api/models/current` 切换默认模型时：

- active provider 存在：更新 cc-evancod managed settings。
- 无 active provider：更新普通用户 settings。

## 多 provider/model 会话兼容

桌面端支持「全局默认 provider」和「会话级 runtime 选择」两层模型选择。

### 前端选择入口

`desktop/src/components/controls/ModelSelector.tsx` 构建 provider/model 下拉列表：

- Claude 官方模型。
- OpenAI 官方模型。
- 所有保存的自定义 provider。
- 每个自定义 provider 展示其 `main/haiku/sonnet/opus` 模型映射，重复 model id 会合并展示。

会话级选择结构是 `RuntimeSelection`：

```ts
{
  providerId: string | null,
  modelId: string,
  effortLevel?: EffortLevel,
}
```

其中：

- `providerId === null` 表示 Claude 官方模式。
- `providerId === 'openai-official'` 表示 OpenAI 官方 OAuth 模式。
- 其他字符串表示保存的自定义 provider ID。

`desktop/src/stores/sessionRuntimeStore.ts` 将会话选择持久化到 `localStorage` 的 `cc-evancod-session-runtime`。

### WebSocket 同步

客户端连接会话时，`desktop/src/stores/chatStore.ts` 会读取当前 session 的 runtime selection，并发送：

```json
{ "type": "set_runtime_config", "providerId": "...", "modelId": "...", "effortLevel": "..." }
```

用户在模型选择器中切换 provider/model 时，也会发送同样的 `set_runtime_config`。

服务端消息类型定义在 `src/server/ws/events.ts`。

### 服务端 runtime override

`src/server/ws/handler.ts` 中的 `handleSetRuntimeConfig()` 做以下处理：

1. 校验 `modelId` 非空。
2. 校验 `effortLevel` 属于 `low/medium/high/max`。
3. 写入内存中的 `runtimeOverrides`。
4. 持久化到 session metadata：`runtimeProviderId`、`runtimeModelId`、`effortLevel`。
5. 如果 CLI 还没启动，只记录配置。
6. 如果 CLI 已启动，重启该 session 的 CLI 子进程。
7. 如果当前 turn 正在生成，延迟到当前 turn 完成后再重启。

这保证了模型切换不会中断正在执行的工具或生成中的回合。

### CLI 启动参数与环境

`ConversationService.startSession()` 启动每个桌面 session 对应的 CLI 子进程。

运行时配置通过两条路径传入 CLI：

1. CLI 参数：`--model <modelId>`、`--effort <level>`、`--thinking <mode>`。
2. 子进程 env：由 `buildChildEnv()` 构造 provider 相关 `ANTHROPIC_*` / OAuth env。

`buildChildEnv()` 的关键行为：

- 获取终端 shell 环境，但删除可能冲突的 provider env。
- 如果有显式 `providerId`，通过 `getProviderRuntimeEnv()` 构造该 provider 的会话级 env。
- 如果显式选择了 `model`，覆盖 `ANTHROPIC_MODEL`。
- 官方 OAuth 模式注入 OAuth token。
- 自定义 provider 模式设置 `CLAUDE_CODE_PROVIDER_MANAGED_BY_HOST=1`。
- 设置 `CC_EVANCOD_SKIP_DOTENV=1`，避免项目 `.env` 重新引入旧 provider 配置。
- 设置流式 watchdog、idle timeout、禁用 non-streaming fallback 等桌面端稳定性 env。

### 会话隔离效果

因为显式 provider 会生成 `/proxy/providers/:providerId`，所以不同桌面会话可以同时使用不同协议和不同上游：

- 会话 A：Claude 官方 OAuth。
- 会话 B：Anthropic 兼容第三方服务，直连 `/v1/messages`。
- 会话 C：OpenAI Chat provider，经 `/proxy/providers/<id>/v1/messages` 转到 `/v1/chat/completions`。
- 会话 D：OpenAI Responses provider，经 `/proxy/providers/<id>/v1/messages` 转到 `/v1/responses`。

每个会话都有独立 CLI 子进程，provider/model 切换通过重启对应 session 子进程生效，不影响其他会话。

## 鉴权策略

Anthropic 直连 provider 支持多种 `authStrategy`，在 `buildProviderAuthEnv()` 和 `buildAnthropicAuthHeaders()` 中分别用于运行时 env 与测试请求：

| 策略                       | 运行时/测试行为                                        |
| -------------------------- | ------------------------------------------------------ |
| `api_key`                  | 使用 `ANTHROPIC_API_KEY` 或 `x-api-key`                |
| `auth_token`               | 使用 `ANTHROPIC_AUTH_TOKEN` 或 `Authorization: Bearer` |
| `auth_token_empty_api_key` | 设置空 `ANTHROPIC_API_KEY` 并使用 token                |
| `dual_same_token`          | 同时设置 api key 和 bearer token，值相同               |
| `dual_dummy`               | api key 与 bearer token 都使用 `dummy`                 |

OpenAI 格式 provider 由本地 proxy 使用真实 `apiKey` 访问上游，CLI 侧只看到 proxy-managed key。

## 设计取舍

- CLI 统一使用 Anthropic Messages API，减少 agent-loop 对多协议的感知。
- 协议差异集中在 server proxy，便于测试和 trace。
- 自定义 provider 与官方 OAuth 分离，避免 Anthropic env 残留干扰官方登录态。
- 全局 active provider 用于默认行为；会话级 runtime selection 用于多会话并行切换。
- provider env 写入 cc-evancod 专属 settings，避免直接改写用户原始 Claude Code settings。
- OpenAI 格式 provider 的真实 key 留在 server 侧，CLI 只访问本地 proxy。
