# 上下文压缩（Context Compaction）实现文档

> 本文档抽取自 Claude Code Evancod 项目的上下文压缩体系，目标是让另一个团队（VSCode 插件，Vue3 + TS + Node）能够理解并复刻这套机制。
> 文档聚焦「**自动压缩（auto-compact）**」这条主线，同时说明其他几种配套压缩策略，便于选择性实现。

---

## 0. 名词与总览

一个 Agent 会话本质上是一个 **消息数组**（`Message[]`），每轮对话把新的 user / assistant / tool_result 消息追加进去，再整体发给大模型。消息越积越多，早晚会超过模型的上下文窗口（context window）。上下文压缩就是在「快撑爆之前」把历史消息**摘要化**或**裁剪**，腾出空间继续对话。

本项目里有 **5 种压缩机制**，从轻到重排列：

| 机制 | 触发方式 | 做什么 | 是否调用大模型 | 是否有损 |
|------|---------|--------|--------------|---------|
| **Tool-Result Budget** | 每轮请求前 | 单条 tool_result 超预算时截断内容 | 否 | 轻度有损 |
| **Microcompact（微压缩）** | 每轮请求前 | 清除旧的工具调用结果（Read/Bash/Grep 等），保留最近 N 个 | 否 | 中度有损（只删工具结果） |
| **Snip** | 每轮请求前 | 裁剪历史消息尾部（实验特性） | 否 | 有损 |
| **Auto-Compact（自动压缩）★** | Token 超阈值时 | 用大模型把整段历史生成结构化摘要，替换原始消息 | **是** | 有损但保留语义 |
| **Context Collapse** | 90%/95% 阈值 | 更细粒度的分段折叠（实验特性） | 是 | 有损 |

**★ 你要复刻的核心是 Auto-Compact。** 其余是优化项，理解主线后可选做。

在一次 query 循环的**每一轮**里，这些机制的执行顺序是（见 `src/query.ts`）：

```
applyToolResultBudget   →   snip   →   microcompact   →   contextCollapse   →   autocompact
        (预算截断)          (裁剪)      (删工具结果)         (折叠)             (摘要压缩)
```

设计意图：**先用便宜的手段（删工具结果、折叠）把 token 压下去，如果压下去后没到自动压缩阈值，就不需要触发昂贵的大模型摘要。**

## 1. Auto-Compact 主线逻辑

对应源码：
- `src/services/compact/autoCompact.ts`（阈值计算 + 触发判定 + 编排）
- `src/services/compact/compact.ts`（实际压缩执行）
- `src/services/compact/prompt.ts`（摘要提示词）

### 1.1 关键常量与阈值

```ts
// 为压缩摘要的“输出”预留的 token 数（基于 p99.99 摘要输出 = 17,387 token）
const MAX_OUTPUT_TOKENS_FOR_SUMMARY = 20_000

// 各类缓冲区
const AUTOCOMPACT_BUFFER_TOKENS       = 13_000  // 自动压缩缓冲
const WARNING_THRESHOLD_BUFFER_TOKENS = 20_000  // 警告阈值缓冲
const ERROR_THRESHOLD_BUFFER_TOKENS   = 20_000  // 错误阈值缓冲
const MANUAL_COMPACT_BUFFER_TOKENS    = 3_000   // 手动压缩缓冲

// 连续失败 3 次后触发“断路器”，本会话不再尝试自动压缩
const MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES = 3
```

### 1.2 有效上下文窗口

```ts
// 有效窗口 = 模型上下文窗口 - 为摘要输出预留的 token
function getEffectiveContextWindowSize(model: string): number {
  const reserved = Math.min(getMaxOutputTokensForModel(model), 20_000)
  let contextWindow = getContextWindowForModel(model)  // 例如 200_000

  // 可用环境变量覆盖窗口大小（方便测试）
  const override = process.env.CLAUDE_CODE_AUTO_COMPACT_WINDOW
  if (override) { /* parseInt 覆盖 */ }

  return contextWindow - reserved
}
```

### 1.3 自动压缩阈值

```ts
function getAutoCompactThreshold(model: string): number {
  const effective = getEffectiveContextWindowSize(model)
  // 触发阈值 = 有效窗口 - 13k 缓冲
  const threshold = effective - AUTOCOMPACT_BUFFER_TOKENS

  // 可用百分比环境变量覆盖（测试用）
  const pct = process.env.CLAUDE_AUTOCOMPACT_PCT_OVERRIDE
  if (pct) { return Math.min(Math.floor(effective * pct/100), threshold) }

  return threshold
}
```

举例：200k 窗口的模型，`effective = 200k - 20k = 180k`，`threshold = 180k - 13k = 167k`。**当前 token 使用量 ≥ 167k 时触发自动压缩。**

### 1.4 Token 统计（如何知道“用了多少”）

关键函数：`tokenCountWithEstimation(messages)`（`src/utils/tokens.ts`）。

策略是**从消息数组尾部往前找最近一条带真实 API usage 的 assistant 消息**，用它的 `usage.input_tokens` 等作为权威值；对该消息之后的新消息用本地估算补上。这样既准确又不用每轮都调 token count API。

- `getTokenUsage(message)`：取 assistant 消息里的真实 usage（跳过合成消息）。
- `tokenCountFromLastAPIResponse`：取最后一条 API 响应的 token 数。
- 本地估算：`roughTokenCountEstimation`，约 **4 字符 = 1 token**。

> **给 Node/Vue 项目的建议**：你可以用 `@anthropic-ai/tokenizer` 或简单的 `字符数/4` 估算。核心是「优先信任 API 返回的真实 usage，本地估算只补最近几条新消息」。

### 1.5 是否应该压缩：`shouldAutoCompact`

```ts
async function shouldAutoCompact(messages, model, querySource, snipTokensFreed=0): Promise<boolean> {
  // 1) 递归守卫：压缩本身/记忆抽取是“分叉 agent”，不能再触发压缩，否则死锁
  if (querySource === 'session_memory' || querySource === 'compact') return false

  // 2) 用户是否禁用（环境变量 / 配置）
  if (!isAutoCompactEnabled()) return false

  // 3) 计算当前 token（减去 snip 已释放的量）
  const tokenCount = tokenCountWithEstimation(messages) - snipTokensFreed
  const { isAboveAutoCompactThreshold } = calculateTokenWarningState(tokenCount, model)
  return isAboveAutoCompactThreshold
}
```

`isAutoCompactEnabled()` 的判定：
```ts
function isAutoCompactEnabled(): boolean {
  if (isEnvTruthy(process.env.DISABLE_COMPACT)) return false
  if (isEnvTruthy(process.env.DISABLE_AUTO_COMPACT)) return false
  return getGlobalConfig().autoCompactEnabled  // 用户配置
}
```

### 1.6 Token 警告状态（供 UI 显示进度条）

`calculateTokenWarningState(tokenUsage, model)` 返回：

```ts
{
  percentLeft,                 // 剩余百分比（给 UI 显示 "还剩 X%"）
  isAboveWarningThreshold,     // 超过警告阈值（提醒用户快满了）
  isAboveErrorThreshold,       // 超过错误阈值
  isAboveAutoCompactThreshold, // 超过自动压缩阈值 → 触发压缩
  isAtBlockingLimit,           // 到达阻塞上限（连手动都压不下去，禁止再发）
}
```

各阈值关系（tokenUsage 从小到大）：
```
warningThreshold = threshold - 20k
errorThreshold   = threshold - 20k
autoCompactThreshold = threshold        ← 触发自动压缩
blockingLimit = effectiveWindow - 3k    ← 硬上限
```

> **给 Node/Vue 项目**：这一块直接映射到你的「上下文用量指示器」UI。前端订阅 `percentLeft` 显示环形/条形进度，后端在每轮返回这些布尔标志。

### 1.7 编排：`autoCompactIfNeeded`

这是 query 循环每轮调用的入口：

```ts
async function autoCompactIfNeeded(messages, toolUseContext, cacheSafeParams, querySource, tracking, snipTokensFreed) {
  if (isEnvTruthy(process.env.DISABLE_COMPACT)) return { wasCompacted: false }

  // 断路器：连续失败 >= 3 次就不再试
  if (tracking?.consecutiveFailures >= 3) return { wasCompacted: false }

  const model = toolUseContext.options.mainLoopModel
  if (!(await shouldAutoCompact(messages, model, querySource, snipTokensFreed)))
    return { wasCompacted: false }

  const recompactionInfo = { /* 记录本次是否是“连续压缩”，用于诊断 */ }

  // 实验：先尝试“会话记忆压缩”（把历史转成持久记忆），失败再走常规压缩
  const sessionMemoryResult = await trySessionMemoryCompaction(...)
  if (sessionMemoryResult) { /* 清理 + 返回 */ }

  try {
    const compactionResult = await compactConversation(
      messages, toolUseContext, cacheSafeParams,
      true,        // suppressFollowUpQuestions（自动压缩不问用户）
      undefined,   // 无自定义指令
      true,        // isAutoCompact
      recompactionInfo,
    )
    setLastSummarizedMessageId(undefined)
    runPostCompactCleanup(querySource)  // 重置文件缓存等状态
    return { wasCompacted: true, compactionResult, consecutiveFailures: 0 }
  } catch (error) {
    // 失败计数 +1，达 3 次触发断路器
    const next = (tracking?.consecutiveFailures ?? 0) + 1
    return { wasCompacted: false, consecutiveFailures: next }
  }
}
```

**关键设计点：**
- **断路器**：曾统计到 1279 个会话连续失败 50+ 次，一天浪费 25 万次 API 调用。3 次失败后本会话放弃自动压缩。
- **自动压缩不弹错误通知**：因为下一轮会重试，弹错反而困惑用户。手动 `/compact` 才弹。

## 2. 压缩执行：`compactConversation`

这是把「一大段历史」变成「一段摘要」的核心函数，源码 `src/services/compact/compact.ts`。流程：

```
1. 计算 preCompactTokenCount（压缩前 token）
2. 执行 PreCompact 钩子（允许用户注入自定义摘要指令）
3. 构造摘要请求：getCompactPrompt(customInstructions) → 一条 user 消息
4. 循环调用大模型生成摘要 streamCompactSummary（含 prompt-too-long 重试）
5. 清空文件读取缓存 readFileState
6. 并行生成“压缩后附件”：
   - 最近读过的文件（重新读，避免模型再 Read）
   - 计划文件 / plan 模式提示
   - 已调用的技能（skills）内容
   - 工具/agent/MCP 的声明增量
7. 执行 SessionStart 钩子
8. 创建 compact 边界标记 boundaryMarker
9. 组装摘要消息 summaryMessages
10. 记录 tengu_compact 事件（遥测）
11. 执行 PostCompact 钩子
12. 返回 CompactionResult
```

### 2.1 返回结构 `CompactionResult`

```ts
interface CompactionResult {
  boundaryMarker: SystemMessage       // 压缩边界标记（分隔“压缩前/后”）
  summaryMessages: UserMessage[]      // 摘要内容（作为 user 消息注入）
  attachments: AttachmentMessage[]    // 压缩后重新注入的附件（文件/技能/计划）
  hookResults: HookResultMessage[]    // SessionStart 钩子产出
  messagesToKeep?: Message[]          // 保留的原始消息（部分压缩时用）
  preCompactTokenCount?: number
  postCompactTokenCount?: number
  truePostCompactTokenCount?: number  // 压缩后真实上下文大小估算
}
```

### 2.2 压缩后消息如何组装

```ts
function buildPostCompactMessages(result: CompactionResult): Message[] {
  return [
    result.boundaryMarker,                                  // 1. 边界标记
    ...result.summaryMessages,                              // 2. 摘要
    ...stripStaleUsageFromPreservedMessages(result.messagesToKeep ?? []), // 3. 保留消息
    ...result.attachments,                                  // 4. 附件
    ...result.hookResults,                                  // 5. 钩子结果
  ]
}
```

> `stripStaleUsageFromPreservedMessages`：把保留下来的 assistant 消息里的 `usage` 归零。因为压缩后旧的 usage 不再代表当前上下文大小，否则 token 计量会被旧值卡住（问题 #743）。

### 2.3 压缩边界标记（重要概念）

压缩不是「删掉旧消息」，而是**在数组里插入一条 `boundaryMarker`，之后发给 API 时只取边界之后的消息**（`getMessagesAfterCompactBoundary`）。这样：
- 原始消息仍保存在磁盘 transcript 里（可回溯）。
- API 请求只看到「边界后的摘要 + 保留消息 + 附件」。

query 循环里：
```ts
let messagesForQuery = [...getMessagesAfterCompactBoundary(messages)]
```

> **给 Node/Vue 项目**：你可以用同样思路——完整历史存磁盘/DB，内存里维护一个「有效发送窗口」。或者简单点：压缩后直接用新数组替换旧数组，把原始存归档文件。

### 2.4 摘要提示词（`prompt.ts` 核心）

摘要要求模型产出**结构化 9 段摘要**，这是复刻质量的关键：

```
1. Primary Request and Intent   （用户的明确请求与意图）
2. Key Technical Concepts        （关键技术概念）
3. Files and Code Sections       （涉及的文件与代码片段，附完整代码）
4. Errors and fixes              （遇到的错误及修复，含用户反馈）
5. Problem Solving               （已解决问题与进行中的排查）
6. All user messages             （所有非工具结果的用户消息）★
7. Pending Tasks                 （待办任务）
8. Current Work                  （压缩前正在做的事，附文件名和代码）
9. Optional Next Step            （下一步，须与用户最近请求直接相关，附原文引用）
```

提示词还包含：
- **NO_TOOLS_PREAMBLE**：强制模型「只输出文本，不要调用任何工具」（放在最前，因为 Sonnet 4.6+ 有时会尝试调工具，浪费唯一一轮）。
- 要求用 `<analysis>...</analysis>` 做草稿思考，再用 `<summary>...</summary>` 输出正式摘要。
- `formatCompactSummary` 会**剥离 `<analysis>`**（草稿无信息价值），把 `<summary>` 替换成可读标题。

摘要最终注入的措辞（`getCompactUserSummaryMessage`）：
```
This session is being continued from a previous conversation that ran out of context.
The summary below covers the earlier portion of the conversation.
<格式化后的摘要>

If you need specific details from before compaction ..., read the full transcript at: <path>

Continue the conversation from where it left off without asking the user any further questions.
Resume directly — do not acknowledge the summary...
```

> 最后一句很重要：告诉模型「别寒暄、别复述摘要、直接接着干」。

### 2.5 生成摘要的两条路径（`streamCompactSummary`）

1. **缓存共享 fork 路径（首选）**：`runForkedAgent`，复用主对话的 prompt cache 前缀（system/tools/messages 前缀相同）→ 省 token。**不设 maxOutputTokens**，否则会破坏 thinking 配置导致缓存失效。
2. **常规流式路径（fallback）**：直接 `queryModelWithStreaming`，系统提示为 `"You are a helpful AI assistant tasked with summarizing conversations."`，`thinkingConfig: disabled`，`maxOutputTokensOverride = min(20k, 模型上限)`。

生成前会 `stripImagesFromMessages`（把图片换成 `[image]` 文本，摘要不需要图且图会撑爆请求）和 `stripReinjectedAttachments`（去掉压缩后会重新注入的技能附件，避免浪费）。

### 2.6 Prompt-Too-Long 重试（CC-1180）

如果摘要请求**本身**就超长，进入 `truncateHeadForPTLRetry`：按「API 轮次分组」丢弃最老的组，直到覆盖 token 缺口（最多重试 3 次，`MAX_PTL_RETRIES`）。丢完还不够就抛 `ERROR_MESSAGE_PROMPT_TOO_LONG`。

分组用 `groupMessagesByApiRound`（`src/services/compact/grouping.ts`）：把消息按「一个 assistant 回合 + 其后的 tool_result」聚成组，保证不会丢半个工具调用对。

## 3. Microcompact（微压缩）—— 便宜的前置优化

源码：`src/services/compact/microCompact.ts`。

**核心思想**：不摘要整段历史，只**清除旧的「可压缩工具」的结果内容**，保留最近 N 个。可压缩工具白名单：

```ts
const COMPACTABLE_TOOLS = new Set([
  'Read', 'Bash'..., 'Grep', 'Glob', 'WebSearch', 'WebFetch', 'Edit', 'Write'
])
```

清除时把 tool_result 内容替换成 `'[Old tool result content cleared]'`。

### 3.1 两个子策略

1. **时间触发（Time-Based）** `maybeTimeBasedMicrocompact`：
   - 若「距上一条 assistant 消息的时间间隔」超过阈值（例如几分钟），说明服务端 prompt cache 已过期，整段前缀反正要重写 → 此时提前清掉旧工具结果，缩小重写量。
   - 直接改消息内容（缓存已冷，无需保留）。
   - 保留最近 `keepRecent` 个（至少 1 个）。

2. **缓存微压缩（Cached MC）** `cachedMicrocompactPath`（ant-only 特性）：
   - 用 **Cache Editing API** 删除工具结果，**不改本地消息**（cache_reference / cache_edits 在 API 层添加），从而不破坏缓存前缀。
   - 用「数量」触发（config.triggerThreshold / keepRecent）。
   - 边界消息延迟到 API 响应后再创建（用真实的 cache_deleted_input_tokens）。

> **给 Node/Vue 项目的建议**：Cache Editing 是 Anthropic 特定能力，非 Anthropic 后端可跳过。**时间触发的内容清除**很好复刻——遍历消息，把除最近 N 个之外的 Read/Bash/Grep 结果内容替换成占位符即可，几乎零成本地省下大量 token。

### 3.2 Token 估算

`estimateMessageTokens` 遍历所有 block：text/tool_result/image(≈2000)/thinking/tool_use，累加后 **× 4/3 保守放大**。图片统一按 2000 token 估。

## 4. Tool-Result Budget（单条结果预算）

query 循环最前面调用 `applyToolResultBudget`：对**单条** tool_result 的聚合大小做上限，超了就截断内容。运行在 microcompact 之前——因为 cached MC 只按 tool_use_id 操作、不看内容，两者互不干扰、可组合。

只对「声明了有限 `maxResultSizeChars` 的工具」生效。

## 5. 部分压缩（Partial Compact）—— 用户手动选点压缩

`partialCompactConversation`：用户在消息选择器里选一个「支点消息」，两种方向：
- `from`：摘要**支点之后**的消息，保留之前的（保留 prompt cache）。
- `up_to`：摘要**支点之前**的消息，保留之后的（缓存失效，因为摘要在保留消息之前）。

保留的消息通过 `boundaryMarker.compactMetadata.preservedSegment` 记录 head/anchor/tail UUID，加载时重新链接消息父子关系。这是自动压缩的手动增强版，复刻时可作为进阶功能。

## 6. query 循环中的集成点（`src/query.ts`）

每轮循环里，压缩的编排代码（简化）：

```ts
let messagesForQuery = [...getMessagesAfterCompactBoundary(messages)]
let tracking = autoCompactTracking

// 1) 单条 tool_result 预算截断
messagesForQuery = await applyToolResultBudget(messagesForQuery, ...)

// 2) snip（实验）
let snipTokensFreed = 0
if (feature('HISTORY_SNIP')) {
  const snipResult = snipModule.snipCompactIfNeeded(messagesForQuery)
  messagesForQuery = snipResult.messages
  snipTokensFreed = snipResult.tokensFreed
}

// 3) microcompact（删旧工具结果）
const microcompactResult = await deps.microcompact(messagesForQuery, toolUseContext, querySource)
messagesForQuery = microcompactResult.messages

// 4) context collapse（实验）
if (feature('CONTEXT_COLLAPSE')) {
  const collapseResult = await contextCollapse.applyCollapsesIfNeeded(messagesForQuery, ...)
  messagesForQuery = collapseResult.messages
}

// 5) autocompact（大模型摘要）★
const { compactionResult, consecutiveFailures } = await deps.autocompact(
  messagesForQuery, toolUseContext,
  { systemPrompt, userContext, systemContext, toolUseContext, forkContextMessages: messagesForQuery },
  querySource, tracking, snipTokensFreed,
)

if (compactionResult) {
  // 记录成功事件、更新 tracking
  tracking = { compacted: true, turnId: uuid(), turnCounter: 0, consecutiveFailures: 0 }
  const postCompactMessages = buildPostCompactMessages(compactionResult)
  for (const message of postCompactMessages) yield message  // 推送给 UI
  messagesForQuery = postCompactMessages  // 用压缩后消息继续本轮
} else if (consecutiveFailures !== undefined) {
  // 失败：更新断路器计数
  tracking = { ...(tracking ?? {...}), consecutiveFailures }
}
```

### 6.1 跨轮状态 `AutoCompactTrackingState`

```ts
type AutoCompactTrackingState = {
  compacted: boolean          // 本会话是否已压缩过
  turnCounter: number         // 距上次压缩过了几轮
  turnId: string              // 每轮唯一 ID
  consecutiveFailures?: number // 连续失败计数（断路器）
}
```

这个状态在 query 循环的 `State` 里通过每次迭代传递（`src/query.ts` 的 `next: State = { ..., autoCompactTracking: tracking }`）。

## 7. 复刻到 Vue3 + TS + Node 的落地建议

### 7.1 必做的核心（对应 Auto-Compact）

1. **消息模型**：定义 `Message`（user / assistant / tool_result / system-boundary）联合类型。
2. **Token 统计模块**：
   - 优先读 API 返回的 `usage.input_tokens` 作为权威值。
   - 本地用 `字符数 / 4` 估算补新消息（或用 `@anthropic-ai/tokenizer`）。
3. **阈值计算**：
   ```
   effectiveWindow = contextWindow - 摘要预留(20k)
   autoCompactThreshold = effectiveWindow - 13k
   触发条件：currentTokens >= autoCompactThreshold
   ```
4. **压缩执行**：
   - 构造 9 段结构化摘要提示词（照抄 §2.4，这是质量关键）。
   - 用 `thinking: disabled`、限定输出 token、系统提示设为“你是对话摘要助手”。
   - 生成前剥离图片（换成 `[image]`）。
   - 摘要注入措辞照抄 §2.4 末尾（“别寒暄、直接继续”）。
5. **边界机制**：压缩后 = `[边界标记, 摘要消息, (可选保留消息), 附件]`，原始历史归档到磁盘/DB。
6. **断路器**：连续失败 3 次停止自动压缩。
7. **UI 指示器**：暴露 `percentLeft` / `isAboveWarningThreshold` 给前端。

### 7.2 强烈建议做的优化（对应 Microcompact）

- 在触发昂贵的大模型摘要**之前**，先跑一遍「删旧工具结果」：保留最近 N 个 Read/Bash/Grep 结果，其余内容替换成 `[Old tool result content cleared]`。往往能把 token 压到阈值以下，避免摘要调用。

### 7.3 VSCode 插件架构映射

- **Node 后端（extension host）**：跑 query 循环、token 统计、压缩编排、调 Anthropic API。压缩是 CPU/IO 轻、网络重的操作，放后端合适。
- **Vue3 前端（webview）**：显示上下文用量进度条（订阅 `percentLeft`），显示压缩边界（"已从之前的对话继续"），显示压缩进行中的 loading 状态（对应本项目的 `onCompactProgress` 事件：`hooks_start` / `compact_start` / `compact_end`）。
- **进度事件**：本项目通过 `context.onCompactProgress?.({ type: 'compact_start' })` 等回调驱动 UI。你可以用 WebSocket / postMessage 把这些阶段推给 webview。

### 7.4 可选进阶

- **部分压缩**（用户选点压缩）：对应 §5。
- **文件/技能重注入**：压缩后把最近读过的文件重新读进来（§2 第 6 步），避免模型重复 Read。这对编码 agent 体验提升明显。
- **PreCompact / PostCompact 钩子**：允许用户注入自定义摘要指令。

## 9. 一张流程图（Auto-Compact 主线）

```
每轮 query 开始
   │
   ├─ messagesForQuery = getMessagesAfterCompactBoundary(所有消息)
   ├─ applyToolResultBudget（截断超大单条结果）
   ├─ [snip]（裁剪尾部，实验）
   ├─ microcompact（删旧 Read/Bash 结果，保留最近 N 个）
   ├─ [contextCollapse]（分段折叠，实验）
   │
   ├─ autoCompactIfNeeded:
   │     ├─ 断路器检查（失败≥3 → 跳过）
   │     ├─ shouldAutoCompact?
   │     │     tokenCount = tokenCountWithEstimation(messages) - snipFreed
   │     │     return tokenCount >= autoCompactThreshold
   │     │
   │     └─ 若需要压缩:
   │           ├─ [trySessionMemoryCompaction]（实验，优先）
   │           └─ compactConversation:
   │                 ├─ PreCompact 钩子
   │                 ├─ streamCompactSummary（大模型生成 9 段摘要）
   │                 │     ├─ 剥离图片
   │                 │     ├─ fork 缓存共享路径（首选）
   │                 │     ├─ 常规流式路径（fallback）
   │                 │     └─ PTL 超长 → truncateHeadForPTLRetry（最多 3 次）
   │                 ├─ 清空 readFileState
   │                 ├─ 生成附件（最近文件/技能/计划）
   │                 ├─ SessionStart 钩子
   │                 ├─ 创建 boundaryMarker + summaryMessages
   │                 └─ PostCompact 钩子
   │
   ├─ 若压缩成功:
   │     messagesForQuery = buildPostCompactMessages(result)
   │                      = [边界, 摘要, 保留消息, 附件, 钩子结果]
   │     更新 tracking（compacted=true, consecutiveFailures=0）
   │
   └─ 用 messagesForQuery 发起本轮 API 请求
```

---

**总结一句话**：核心是「**每轮请求前，先用便宜手段（截断/删工具结果）压 token，压不下去再用大模型把整段历史生成一份 9 段结构化摘要，用摘要+边界标记替换原始消息继续对话**」，配合 token 阈值判定、断路器、图片剥离、prompt-too-long 重试等工程细节保证鲁棒性。
